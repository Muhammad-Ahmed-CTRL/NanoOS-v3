COMMAND 1: calc (Calculator App)
FILE: Apps/calculator.asm
================================================================

WHAT IT DOES:
User types things like "15 + 7" or "100 / 4" and the OS
computes and shows the result. Supports +, -, *, /

HOW THE CODE WORKS (explain step by step):

STEP 1 — User types expression:
  The shell reads the input into 'sh_input_buf' using sh_read_line.
  Example: User types "12 + 8" and presses Enter.

STEP 2 — Parsing first number:
  Function calc_parse_int is called. It reads character by
  character from the buffer. For each digit '0' to '9', it:
  a) Subtracts '0' from the ASCII byte to get actual digit value
  b) Multiplies running total by 10:  imul eax, eax, 10
  c) Adds the new digit:              add eax, edx
  It also checks for a leading '-' sign for negative numbers.

STEP 3 — Reading the operator:
  After the first number, calc_skip_spaces skips any spaces.
  Then: lodsb   (load byte from [ESI] into AL, increment ESI)
  AL now holds '+', '-', '*', or '/'

STEP 4 — Parsing second number:
  calc_parse_int is called again for the second number.

STEP 5 — Performing the math:
  The code compares the operator with each symbol:
    cmp bl, '+'    je .add     -> add eax, ecx
    cmp bl, '-'    je .sub     -> sub eax, ecx
    cmp bl, '*'    je .mul     -> imul eax, ecx   (signed multiply)
    cmp bl, '/'    je .div     -> cdq             (sign extend)
                                  idiv ecx        (signed divide)
  For division, CDQ sign-extends EAX into EDX:EAX pair,
  then IDIV divides that 64-bit value by ECX.
  Quotient goes to EAX, remainder goes to EDX.

STEP 6 — Showing the result:
  The result in EAX is printed using sh_print_int.
  If it was division, the remainder in EDX is also shown.

VIVA QUESTIONS FOR SAMI (calc):

Q: What instruction did you use for multiplication?
A: IMUL (Integer Multiply) — it handles signed numbers.
   For example: imul eax, ecx  multiplies EAX by ECX and
   stores the result back in EAX.

Q: How does division work in your code?
A: I use IDIV for signed division. Before calling IDIV,
   I must call CDQ which "sign extends" EAX into the
   EDX:EAX register pair (makes it 64-bit). Then:
   IDIV ECX gives EAX = quotient and EDX = remainder.

Q: What happens if the user divides by zero?
A: Before calling IDIV, I check: test ecx, ecx / jz .div0
   If ECX is zero, I jump to the error handler which prints
   "Error: Division by zero." and goes back to the prompt.

Q: How do you parse a negative number like -5?
A: I check if the first character is '-'. If yes, I set
   a flag (ECX = 1), skip past it, parse the positive
   digits, then negate the result: neg eax

COMMAND 2: game (Number Guessing Game)
FILE: Apps/game.asm
================================================================

WHAT IT DOES:
OS secretly picks a number from 1 to 99. User guesses.
OS replies "Too LOW", "Too HIGH", or "CORRECT!".
After winning, it shows rank: S, A, B, or C based on attempts.

HOW THE CODE WORKS:

STEP 1 — Generating the secret number (the cool part!):
  Function game_random_1_99 does this with hardware:
  
  First, latch the PIT (Programmable Interval Timer):
    mov al, 0x00
    out 0x43, al        ; Send latch command to PIT
  
  Then read 2 bytes from PIT channel 0:
    in  al, 0x40        ; Low byte of counter
    in  al, 0x40        ; High byte of counter
  
  Combine into 16-bit value:  or eax, ecx
  
  Mix with LCG (Linear Congruential Generator):
    imul eax, eax, 1664525
    add  eax, 1013904223
  
  XOR with previous secret for extra randomness:
    xor eax, [game_secret]
  
  Ensure positive then MOD 99:
    neg eax (if negative)
    xor edx, edx
    mov ecx, 99
    div ecx             ; EDX = remainder (0 to 98)
    lea eax, [edx + 1]  ; Add 1 to get range 1..99

STEP 2 — Reading the user's guess:
  sh_read_line reads input into sh_input_buf.
  calc_parse_int parses the number. Result in ECX.

STEP 3 — Comparing with secret:
  mov eax, [game_secret]
  cmp ecx, eax
  je  .correct    ; Equal? Player wins!
  jl  .too_low    ; Less than? Too low
  jg  .too_high   ; Greater? Too high

STEP 4 — Ranking system:
  After winning:
  mov eax, [game_attempts]
  cmp eax, 3   jle .rank_s   ; 3 or fewer = S rank
  cmp eax, 7   jle .rank_a   ; 4-7 = A rank
  cmp eax, 12  jle .rank_b   ; 8-12 = B rank
  ; More than 12 = C rank

VIVA QUESTIONS FOR SAMI (game):

Q: How did you generate random numbers without a library?
A: I used hardware! The PIT (Programmable Interval Timer)
   chip counts down at 1,193,182 times per second. By reading
   port 0x40 at the exact moment the user starts the game,
   the value is always different — giving us a "random" seed.
   I then mix it with an LCG algorithm using IMUL and ADD to
   spread the bits evenly, and use DIV to get a number in range.

Q: What is the IN instruction?
A: IN is a special x86 instruction for reading from hardware
   ports. "in al, 0x40" reads one byte from port 0x40 (PIT
   channel 0) directly into the AL register. You cannot use
   regular MOV for hardware ports.

Q: How do you get MOD (remainder)?
A: After the DIV instruction, the remainder is automatically
   stored in EDX. So "div ecx" with ecx=99 puts the result
   in EAX and the remainder (0 to 98) in EDX.

   The low byte (0x20) is the ASCII code for space.

KERNEL COMPONENT 1: GDT and Protected Mode Switch
FILE: kernel/kernel.asm (lines 88-131)
================================================================

WHAT IS THE GDT?
The Global Descriptor Table tells the CPU how memory is
organized in Protected Mode. It has 3 entries:
  - gdt_null: Mandatory empty first entry (8 zero bytes)
  - gdt_code: Defines the code segment (execute permission)
  - gdt_data: Defines the data segment (read/write permission)

HOW THE SWITCH HAPPENS:
  STEP 1: Enable A20 line (allows access above 1MB):
    in al, 0x92         ; Read system control port
    or al, 0x02         ; Set bit 1 = enable A20
    out 0x92, al        ; Write back

  STEP 2: Load GDT into CPU:
    lgdt [gdt_descriptor]   ; LGDT = Load GDT Register

  STEP 3: Set bit 0 of CR0 register:
    mov eax, cr0
    or  eax, 0x1        ; Bit 0 = Protected Mode Enable
    mov cr0, eax

  STEP 4: Far jump to flush CPU pipeline:
    jmp CODE_SEG:pm32_entry
    This jump flushes the instruction pipeline and switches
    the CPU to 32-bit mode permanently.

VIVA QUESTIONS FOR HANAN (GDT/PM):

Q: Why do you need the GDT?
A: In 32-bit Protected Mode, the CPU uses segment selectors
   to look up memory permissions in the GDT. Unlike Real Mode
   where segment*16+offset is used, Protected Mode enforces
   access rights. We define a flat memory model: both code
   and data segments span the full 4GB address space.

Q: What is the A20 line?
A: The A20 address line is bit 20 of the memory bus. In old
   Real Mode machines, it was disabled for compatibility.
   We must enable it using port 0x92 to access memory above
   1MB in Protected Mode.

Q: What is CR0 and why do you modify it?
A: CR0 is a special x86 Control Register. Bit 0 (PE bit)
   controls whether the CPU is in Real Mode (0) or Protected
   Mode (1). When we set it with "or eax, 0x1" and "mov cr0,
   eax", the CPU immediately switches modes.

KERNEL COMPONENT 2: IDT (Interrupt Descriptor Table)
FILE: kernel/idt.asm
================================================================

WHAT IS THE IDT?
The IDT is a table of 256 entries that tells the CPU what
function to run when an interrupt occurs. It sits at address
0x1000 in memory.

HOW IT WORKS:
Each entry (called a "gate") is 8 bytes containing:
  - The address of the handler function (split into low/high)
  - The code segment selector (0x0008)
  - Type flags (0x8E00 = Present, Ring0, 32-bit interrupt gate)

Setting a gate (idt_set_gate function):
  ; Low DWORD: Offset[15:0] | (Selector << 16)
  mov eax, edx           ; EDX = handler address
  and eax, 0x0000FFFF    ; Keep only low 16 bits
  mov ebx, 0x00080000    ; Code segment = 0x08
  or  eax, ebx
  mov [edi], eax         ; Write first 4 bytes

  ; High DWORD: Offset[31:16] | Type | Zero
  mov eax, edx
  and eax, 0xFFFF0000    ; Keep only high 16 bits
  or  eax, 0x00008E00    ; Add type flags
  mov [edi + 4], eax     ; Write second 4 bytes

PIC REMAPPING (very important!):
  By default, x86 maps IRQ0 to interrupt 8 which conflicts
  with CPU exceptions. We remap:
    IRQ0-7  -> Interrupt vectors 0x20-0x27
    IRQ8-15 -> Interrupt vectors 0x28-0x2F

  Code in remap_pic:
    mov al, 0x11  out 0x20, al   ; Init command to Master PIC
    mov al, 0x20  out 0x21, al   ; Map IRQ0 to INT 0x20
    mov al, 0x28  out 0xA1, al   ; Map IRQ8 to INT 0x28
    ...etc

IRQ0 HANDLER (Timer — runs 18.2 times per second):
  irq0:
    pushad                       ; Save ALL registers
    inc dword [tick_counter]     ; Increment our timer
    ; ... matrix screensaver logic ...
    ; ... wake sleeping tasks ...
    mov al, 0x20
    out 0x20, al                 ; Send EOI to PIC (mandatory!)
    popad                        ; Restore registers
    iretd                        ; Return from interrupt

IRQ1 HANDLER (Keyboard):
  irq1:
    pushad
    in al, 0x60                  ; Read scancode from keyboard
    movzx ebx, byte [kb_w_ptr]  ; Get write pointer
    mov [kb_buffer + ebx], al   ; Store in ring buffer
    inc byte [kb_w_ptr]          ; Advance write pointer
    mov al, 0x20
    out 0x20, al                 ; EOI
    popad
    iretd

VIVA QUESTIONS FOR HANAN (IDT):

Q: Why do you remap the PIC?
A: The Intel 8259 PIC ships with IRQ0 mapped to interrupt 8.
   But interrupt 8 is the CPU's "Double Fault" exception.
   This causes conflicts. We reprogram both PIC chips using
   port I/O to remap IRQ0 to vector 0x20 and IRQ8 to 0x28,
   where there are no CPU exceptions.

Q: What is EOI and why is it essential?
A: EOI stands for "End Of Interrupt." After an IRQ handler
   runs, we MUST send 0x20 to port 0x20 to tell the PIC
   the interrupt is handled. If we skip this, the PIC will
   never send another interrupt again, freezing the keyboard
   and timer permanently.

Q: How does the keyboard ring buffer work?
A: We have a 256-byte array (kb_buffer) and two pointers:
   kb_w_ptr (write) and kb_r_ptr (read).
   IRQ1 writes to kb_buffer[kb_w_ptr] and increments kb_w_ptr.
   The shell reads from kb_buffer[kb_r_ptr] and increments kb_r_ptr.
   When both pointers are equal, the buffer is empty.
   Since they are single bytes (0-255), they wrap automatically.

KERNEL COMPONENT 3: Cooperative Multitasking
FILE: kernel/task.asm
================================================================

WHAT IS COOPERATIVE MULTITASKING?
NanoOS runs two tasks: "kernel" and "shell". They take turns
using the CPU. A task voluntarily yields control by calling
sys_yield. This is different from preemptive multitasking
where the OS forces switches.

HOW sys_yield WORKS:
  1. Save current task's ESP into its Task Control Block (TCB):
     mov eax, [current_task]
     imul eax, TCB_SIZE          ; TCB_SIZE = 64 bytes
     add eax, TCB_BASE           ; TCB_BASE = 0x20000
     mov [eax + 8], esp          ; Save stack pointer

  2. Find the next READY task (round-robin search):
     Loop through tasks, find one with state = TS_READY (0)

  3. Switch to it:
     mov esp, [eax + 8]          ; Load its saved stack pointer
     mov [current_task], ebx     ; Update current task ID
     popad
     ret                         ; Return to where it left off

TCB (Task Control Block) structure at 0x20000:
  Offset 0:  task_id   (4 bytes)
  Offset 4:  state     (4 bytes: 0=Ready, 1=Running, 2=Sleeping)
  Offset 8:  esp       (saved stack pointer, 4 bytes)
  Offset 16: tick_count (4 bytes)
  Offset 20: wake_tick  (4 bytes, for sleep)
  Offset 24: name      (16 bytes, task name string)

VIVA QUESTIONS FOR HANAN (multitasking):

Q: How does task switching work at the hardware level?
A: It is pure stack manipulation. Each task has its own stack.
   When we switch, we save ESP (stack pointer register) of
   the current task to memory, then load ESP from the next
   task's TCB. When "ret" executes, the CPU pops the saved
   return address from the new stack and jumps there.

Q: What happens when a task calls sys_yield?
A: The call to sys_yield pushes the return address on the
   stack. Inside sys_yield, PUSHAD saves all registers.
   We then save ESP to the TCB, switch the stack to another
   task's saved ESP, do POPAD (restoring that task's registers),
   and RET which jumps back to where that task last yielded.

SHELL COMPONENT 1: Keyboard Input & Scancode Translation
FILE: Apps/shell.asm (sh_read_line function)
================================================================

PROBLEM: The keyboard sends raw "scancodes" not ASCII characters.
  For example, pressing 'A' sends scancode 0x1E.
  The shift key sends 0x2A.

HOW sh_read_line SOLVES THIS:
  
  1. Wait for a scancode from the ring buffer:
     call sys_get_scancode
     (loops until kb_w_ptr != kb_r_ptr, then reads one byte)

  2. Handle shift key:
     cmp al, 0x2A  je .set_shift    ; Left shift pressed
     cmp al, 0x2A | 0x80  je .clear_shift  ; Left shift released
     ; Break codes have bit 7 set (|0x80)

  3. Ignore break codes (key release events):
     test al, 0x80    ; Check bit 7
     jnz .wait        ; Non-zero = key release, ignore it

  4. Translate scancode to ASCII using lookup table:
     movzx ebx, al              ; EBX = scancode index
     mov al, [sc_table + ebx]   ; sc_table[scancode] = ASCII
     ; If shift held: mov al, [sc_shift + ebx]

  5. Convert uppercase to lowercase:
     cmp al, 'A'  jl .store_char
     cmp al, 'Z'  jg .store_char
     add al, 32   ; A=65, a=97, difference = 32

  6. Handle Backspace (scancode 0x0E maps to ASCII 8):
     Decrement buffer pointer, erase character from VRAM

  7. Handle Enter (scancode 0x1C maps to ASCII 13):
     Null-terminate the buffer, return from sh_read_line

VIVA QUESTIONS FOR AHMED (input):

Q: What is a scancode?
A: A scancode is a raw number sent by the PS/2 keyboard
   controller when a key is pressed or released. It is NOT
   the ASCII code. For example, pressing 'W' sends 0x11,
   not 0x57. We use a lookup table (sc_table) where the
   index is the scancode and the value is the ASCII char.

Q: How do you know if a key was just released?
A: When a key is released, the keyboard sends the same
   scancode but with bit 7 set (value OR 0x80). For example,
   pressing 'A' sends 0x1E, releasing 'A' sends 0x9E.
   We detect this with: test al, 0x80 / jnz .ignore_release

SHELL COMPONENT 2: Command Dispatcher (sh_dispatch)
FILE: Apps/shell.asm (sh_dispatch, line 240)
================================================================

HOW COMMAND MATCHING WORKS:
After the user presses Enter, sh_dispatch is called.

For every command, the code does:
  mov esi, sh_input_buf    ; Pointer to what user typed
  mov edi, cmd_snake_s     ; Pointer to the command string "snake"
  call sh_is_cmd           ; Custom string compare
  je   cmd_snake           ; If matched, jump to that function

sh_is_cmd / sh_strcmp works by comparing byte by byte:
  .loop:
    mov al, [esi]     ; Load byte from input
    mov bl, [edi]     ; Load byte from command string
    cmp al, bl
    jne .not_equal    ; Different? Not a match
    test al, al
    jz .equal         ; Both reached null? Full match!
    inc esi
    inc edi
    jmp .loop

sh_is_cmd also handles "cmd arg" format — it matches even
if the user typed "echo hello" when the command is "echo".

VIVA QUESTIONS FOR AHMED (dispatch):

Q: How does the shell know which command to run?
A: The shell runs a sequential chain of string comparisons.
   For each known command, it calls sh_is_cmd which compares
   the user's input with the stored command string byte by
   byte. The first match triggers a JE (Jump if Equal) to
   that command's handler function.

Q: How does the shell handle unknown commands?
A: At the very end of sh_dispatch, after all known commands
   have been checked without a match, it falls through to
   print the input followed by ": Unknown command" in red.

COMMAND 4: snake (Snake Game)
FILE: Apps/shell.asm (cmd_snake, line ~3800)
================================================================

WHAT IT DOES:
Classic snake game. WASD or arrow keys to move. Eat food (*) to
grow. Hit wall or self = game over. Speed is timer-based.

HOW THE CODE WORKS:

Data Structures:
  snake_body_x  times 256 db 0  ; X positions of each segment
  snake_body_y  times 256 db 0  ; Y positions of each segment
  snake_len     db 5            ; Current length
  snake_dir     db 1            ; 0=up, 1=right, 2=down, 3=left

Movement (move_snake function):
  1. Shift all body segments backwards (tail follows head):
     Loop from tail to head: body[i] = body[i-1]
  2. Move head based on direction:
     dir=0: dec snake_body_y[0]   ; Move up
     dir=1: inc snake_body_x[0]   ; Move right
     dir=2: inc snake_body_y[0]   ; Move down
     dir=3: dec snake_body_x[0]   ; Move left

Collision Detection (check_snake_collision):
  Wall:  if head_x < 1 or head_x > 78 or head_y < 1 or head_y > 22
  Self:  loop through body[1] to body[len-1], check if any
         match head position

Game Speed (snake_delay):
  Uses tick_counter (incremented by IRQ0 timer):
    mov eax, [tick_counter]
    add eax, 3             ; Wait 3 timer ticks
  .wait:
    call sys_yield         ; Yield while waiting
    cmp [tick_counter], eax
    jl .wait               ; Loop until 3 ticks pass

Direction Change (non-blocking key read):
  call sys_get_scancode_noblock  ; Returns 0 if no key
  test al, al
  jz .no_key
  cmp al, 0x48  je .up      ; Scancode 0x48 = Up Arrow
  cmp al, 0x50  je .down    ; Scancode 0x50 = Down Arrow
  cmp al, 0x4B  je .left    ; Scancode 0x4B = Left Arrow
  cmp al, 0x4D  je .right   ; Scancode 0x4D = Right Arrow
  ; Also WASD with scancodes 0x11, 0x1F, 0x1E, 0x20

VIVA QUESTIONS FOR AHMED (snake):

Q: How do you draw the snake without a graphics library?
A: We write directly to VGA memory at 0xB8000. Each character
   cell is 2 bytes. For each snake segment at (x, y):
   offset = y * 160 + x * 2    (160 bytes per row, 2 per char)
   We write the character '#' and color 0x0A (bright green).
   To erase the tail, we write a space (0x20) at the old position.

Q: How is the game speed controlled?
A: We use the tick_counter which is incremented by the IRQ0
   timer (18.2 times per second). The snake_delay function
   records the current tick, adds 3, then loops calling
   sys_yield until the tick_counter reaches that target.
   We reduced from 5 ticks to 3 ticks to make it faster.

COMMAND 5: paint (Pixel Canvas)
FILE: Apps/shell.asm (cmd_paint)
================================================================

WHAT IT DOES:
Full-screen drawing app. Arrow keys move cursor. Number keys
1-8 select color. Space draws a pixel. Backspace erases.

HOW THE CODE WORKS:

  - Instead of letters, we print the FULL BLOCK character (0xDB)

    mov edi, 0xB8000 + offset
    mov byte [edi],   0xDB    ; Full block character
    mov byte [edi+1], color   ; Color attribute

  Reading keys without blocking:
    call sys_get_scancode_noblock  ; Returns 0 if nothing pressed
    The game loop runs continuously, checking for key presses.

VIVA QUESTIONS FOR AHMED (paint):

Q: How do you draw "pixels" in text mode?
A: VGA text mode has 80x25 characters. We treat each character
   cell as a pixel. By writing the full block character (ASCII 219,
   0xDB) with different color attributes, we create colored squares.
   Moving the cursor changes which cell we write to.

COMMAND 6: play (PC Speaker Music)
FILE: Apps/shell.asm (cmd_play)
================================================================

WHAT IT DOES:
Plays a melody using the PC Speaker hardware.

HOW THE CODE WORKS (hardware-level):
  The PC Speaker is controlled by 3 hardware ports:
    Port 0x43: PIT Command Register (configure timer channel 2)
    Port 0x42: PIT Channel 2 Data  (set frequency divisor)
    Port 0x61: System Control Port  (turn speaker on/off)

  To play a note at frequency F Hz:
  STEP 1: Configure PIT Channel 2:
    mov al, 0xB6      ; 10110110b: Channel 2, square wave mode
    out 0x43, al

  STEP 2: Calculate divisor = 1193182 / frequency
    The PIT clock is 1,193,182 Hz. Divisor = PIT_clock / freq
    For 440 Hz (note A): divisor = 1193182 / 440 = 2712

  STEP 3: Send divisor to port 0x42 (low byte then high byte):
    mov al, divisor_low
    out 0x42, al
    mov al, divisor_high
    out 0x42, al

  STEP 4: Turn speaker on by setting bits 0 and 1 of port 0x61:
    in  al, 0x61
    or  al, 0x03      ; Set bits 0 and 1
    out 0x61, al

  STEP 5: Wait (using tick_counter delay)

  STEP 6: Turn speaker off:
    in  al, 0x61
    and al, 0xFC      ; Clear bits 0 and 1
    out 0x61, al

VIVA QUESTIONS FOR AHMED (play):

Q: How do you play sound in an OS with no sound card driver?
A: We use the PC Speaker, a simple buzzer connected directly
   to the CPU via port 0x61. We program the PIT Timer chip
   (port 0x42/0x43) to generate a specific square wave frequency.
   The formula is: divisor = 1,193,182 / desired_frequency_hz.
   We write this divisor to port 0x42 and enable the speaker
   via port 0x61.

COMMAND 7: clock (Digital Clock)
FILE: Apps/shell.asm (cmd_clock)
================================================================

WHAT IT DOES:
Displays a large real-time digital clock. Updates live.
Press any key to exit.

HOW IT READS THE REAL TIME:
  The CMOS Real-Time Clock chip provides the actual time.
  We read it using port I/O:
    out 0x70, al    ; Send register number to CMOS address port
    in  al, 0x71    ; Read value from CMOS data port

  CMOS registers:
    0x00 = seconds
    0x02 = minutes
    0x04 = hours

  BCD Decoding:
  CMOS stores time in BCD (Binary Coded Decimal) format.
  For example, 23 is stored as 0x23, not 0x17.
  Convert: actual_value = (bcd >> 4) * 10 + (bcd & 0x0F)

VIVA QUESTIONS FOR AHMED (clock):

Q: How do you get the real time without an OS API?
A: We read directly from the CMOS RTC (Real-Time Clock) chip
   using two hardware ports: 0x70 (address select) and 0x71
   (data read). The time is stored in BCD format, so we must
   convert: value = high_nibble * 10 + low_nibble.

FILE: Apps/shell.asm and Apps/files.asm

WHAT THEY DO:
NanoOS has a simulated in-memory FAT-style filesystem.
Files and directories are stored in a static table in memory.

HOW ls WORKS:
  Loops through the fs_entries table in memory.
  For each non-empty entry, prints the name.
  Directories shown in green, files in cyan.

HOW cd WORKS:
  Takes the argument after "cd ".
  Searches fs_entries for a matching directory name.
  Updates fs_current_path string if found.

VIVA QUESTIONS FOR AHMED (filesystem):

Q: Is this a real filesystem?
A: No, it is a simulated in-memory filesystem. We maintain
   a table of structs in memory (fs_entries), each containing
   a name, type (file/directory), and contents pointer.
   Commands like mkdir add entries to this table, and ls
   loops through and prints non-empty entries.
